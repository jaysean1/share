During the process, several tools were released, including Vertex AI, Agent Development Kit (ADK), Firebase Studio, and Veo 2. However, what caught my attention the most was the A2A Protocol (Agent-to-Agent Protocol).

The A2A Protocol is an open protocol designed to achieve interoperability and collaboration between AI agents. The A2A protocol allows AI agents built by different vendors or frameworks to communicate with each other, securely exchange information, and coordinate actions, regardless of their underlying technology.

This easily reminds us of the MCP Protocol (Model Context Protocol) released by Anthropic last year, which has recently gained popularity. The MCP protocol is designed by Anthropic to provide a consistent interface between AI agents. The MCP protocol allows AI agents to communicate with each other to achieve collaboration and task distribution. The goal of the MCP protocol is to provide a unified interface for AI agents to promote interoperability and collaboration among them. Moreover, many large internet companies have already publicly implemented the MCP protocol, such as Figma, AutoNavi (Gaode Maps), etc.

So, what is the difference between A2A and MCP? What is their relationship? What can we use it for? Let's explain in detail next.

A real-world example: candidate sourcing

Before diving into complex explanations, let's look at a practical example:

The process of hiring a software engineer can be significantly simplified through A2A collaboration. Within a unified interface (like Agentspace), a user (e.g., a hiring manager) can task their agent to find candidates matching a job listing, location, and skillset. The agent then interacts with other specialized agents to source potential candidates. After receiving these suggestions, the user can direct their agent to schedule further interviews, streamlining the candidate sourcing process. Once the interview process is complete, another agent can be engaged to facilitate background checks. This is just one example of how AI agents need to collaborate across systems to find a qualified job candidate.

How A2A Works

The example above utilizes the A2A protocol to achieve collaboration between multiple agents. Now let's see how this is implemented.

A2A facilitates communication between a "Client Agent" and a "Remote Agent". The Client Agent is responsible for formulating and communicating tasks, while the Remote Agent executes them. This interaction involves several key functions:

Capability Discovery

Agents can advertise their capabilities using JSON-formatted "Agent Cards", enabling Client Agents to identify the most suitable agent for a task.

Task Management

Communication between Client and Remote Agents is oriented towards task completion, with agents working together to fulfill the end-user's request.

Collaboration

Agents can send messages to each other to convey context, replies, artifacts, or user instructions.

User Experience Negotiation

Each message contains "Parts", which are fully formed content fragments, like a generated image. Each part has a specified content type.

Why A2A?

Why do we still need the A2A protocol when we already have the MCP protocol?

Let's look at the limitations of MCP:

  1. MCP Count Limit: The number of MCPs that can be hosted in a single agent is limited (e.g., Cursor's limit is around 40), meaning you cannot add MCPs indefinitely within one agent.
  2. Context Handling Capability Limit: In long-running tasks (like coding, debugging), the performance of a single chat decreases as the context grows, and the model might lose focus or forget previous instructions.
  3. Long Task Tracking Limit: Current MCP struggles with tasks that require waiting for hours or even days (like recruitment, customer service).
  4. Page Rendering Limit: The agent's display rendering needs to happen locally; results from the client-side cannot be obtained via embedding, limiting expressiveness.

Precisely because of these problems and limitations, the A2A protocol emerged. It can effectively solve the above issues, thus enabling the solution described at the beginning. Next, we will try to build a customer service agent ourselves to implement a complete customer service process.

Core Design Principles of A2A

Finally, let's look at the 5 core design principles of A2A:

1. Embrace Agent Capabilities: A2A focuses on enabling agents to collaborate in natural, unstructured ways, even if they don't share memory, tools, and context.

2. Build on Existing Standards: The protocol is built on existing popular standards (HTTP, SSE, JSON-RPC), making it easier to integrate with the IT stacks enterprises already use daily.

3. Secure by Default: Designed to support enterprise-grade authentication and authorization, aligning with OpenAPI's authentication schemes upon release.

4. Support Long-Running Tasks: Designed flexibly to support scenarios ranging from quick tasks to in-depth research that might take hours or even days.

5. Modality Agnostic: A2A supports various modalities, including audio and video streams, not limited to text.

Summary

The A2A protocol provides an open standard for collaboration between agents, thereby enhancing their capabilities and flexibility. The emergence of the A2A protocol can be seen as an evolution of the MCP protocol, better meeting the needs for collaboration among agents.
